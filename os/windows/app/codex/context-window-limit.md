이 메시지는 **Codex가 기억하고 참고해야 할 내용이 한 세션 안에 너무 많이 쌓여서 더 이상 처리하지 못한다**는 뜻입니다. 오류라기보다 **컨텍스트 창 한계**에 도달한 상태입니다. OpenAI 공식 문서에서도 Codex는 여러 환경에서 작업을 이어갈 수 있지만, 각 작업은 모델의 컨텍스트 한계 안에서 처리됩니다. 최신 ChatGPT 릴리스 노트 기준으로 Thinking 모델은 확장된 컨텍스트를 제공하지만, 무제한은 아닙니다. ([OpenAI Help Center][1])

가장 좋은 방법은 **새 스레드로 작업을 분리**하는 것입니다.

## 바로 이렇게 하시면 됩니다

### 1. 현재 Codex 작업을 중단하지 말고, 먼저 요약 파일을 만듭니다

Codex에게 아래 프롬프트를 입력하세요.

```text
현재까지 진행한 작업을 새 Codex 스레드에서 이어갈 수 있도록 요약해 주세요.

반드시 아래 형식으로 작성하세요.

1. 프로젝트 목적
2. 현재 완료된 작업
3. 수정한 파일 목록
4. 생성된 파일 목록
5. 현재 성공한 실행 결과
6. 아직 남은 작업
7. 다음 Codex 스레드에서 가장 먼저 해야 할 일
8. 주의해야 할 제약 조건
9. 사용해야 하는 명령어
10. 다음 스레드에 그대로 붙여 넣을 프롬프트

가능하면 README-CODEX-HANDOFF.md 파일로 저장해 주세요.
```

### 2. 새 스레드를 시작합니다

새 Codex thread 또는 새 conversation을 만들고, 첫 메시지에 아래처럼 넣습니다.

```text
이전 Codex 세션의 컨텍스트가 가득 차서 새 스레드에서 이어갑니다.

먼저 아래 파일을 읽고 현재 상태를 파악하세요.

- README-CODEX-HANDOFF.md
- README.md
- .env.example
- package.json 또는 requirements.txt
- 현재 작업 대상 소스 파일

중요:
이전 대화 내용을 추측하지 말고, 저장소의 실제 파일 상태를 기준으로 판단하세요.
먼저 변경하지 말고 현재 구조를 요약한 뒤, 다음 작업 계획을 제시하세요.
```

### 3. 긴 로그와 스크린샷은 새 스레드에 모두 붙이지 않습니다

이번 PLURA-XDR 자동화 작업처럼 화면 캡처, 로그, 실행 결과가 많으면 Codex 컨텍스트를 빠르게 소모합니다.

새 스레드에는 아래만 넣는 것이 좋습니다.

```text
목표:
PLURA-XDR 대시보드에서 MITRE ATT&CK 탐지 결과를 분석하고,
분석 | 로그 | 포렌식 화면을 확인한 뒤,
이전/현재 상태 diff 기반으로 최종 분석 결과를 Gmail 발송 가능한 JSON/메일 형식으로 정리한다.

현재 상태:
- 로그인 자동화 완료
- MITRE ATT&CK 메뉴 이동 완료
- 위험도 최상 필터 적용 완료
- 오늘 날짜 기준 검색 완료
- 탐지 행 상세 진입 완료
- 원본 로그 캡처 완료
- 포렌식 화면 캡처 일부 완료

다음 작업:
1. 분석 탭 내용 추출
2. 로그 탭 원본 로그 추출
3. 포렌식 탭 이전/현재 값 diff 추출
4. Host IP, 탐지 필터명, 위험도, 분석 결과 정리
5. Gmail 발송용 payload 생성
6. 테스트 실행 결과 저장

주의:
- 실제 화면 기준으로만 판단
- 예시 HostIP 사용 금지
- 날짜는 KST 기준 오늘을 UTC 검색식으로 변환
- 결과는 JSON과 사람이 읽을 수 있는 Markdown 둘 다 생성
```

## 앞으로는 작업을 이렇게 쪼개는 것이 좋습니다

이번 작업은 한 Codex 스레드에서 너무 많은 일을 시켰습니다.

좋은 분리 방식은 다음과 같습니다.

| 스레드      | 목적                        |
| -------- | ------------------------- |
| Thread 1 | 로그인, 메뉴 이동, 날짜 필터, 위험도 필터 |
| Thread 2 | MITRE ATT&CK 탐지 목록 수집     |
| Thread 3 | 상세 화면의 분석/로그/포렌식 데이터 수집   |
| Thread 4 | diff 분석 및 위험도 요약          |
| Thread 5 | Gmail 발송 payload 작성       |
| Thread 6 | 전체 E2E 테스트 및 README 정리    |

즉, **한 스레드 = 하나의 기능 단위**로 가져가야 합니다.

## Codex에 바로 줄 수 있는 복구 프롬프트

아래를 그대로 사용하시면 됩니다.

```text
이전 Codex 세션이 context window 초과로 중단되었습니다.
이제 새 스레드에서 이어서 작업합니다.

절대 이전 대화 내용을 추측하지 말고, 현재 저장소의 실제 파일 상태만 기준으로 분석하세요.

먼저 다음을 수행하세요.

1. 저장소 구조를 확인하세요.
2. README.md, README-CODEX-HANDOFF.md, .env.example 파일이 있으면 읽으세요.
3. package.json 또는 requirements.txt를 확인하여 실행 방식을 파악하세요.
4. 현재까지 구현된 Playwright/Codex 자동화 코드를 확인하세요.
5. 변경하지 말고 현재 상태를 먼저 요약하세요.

그 다음 아래 목표를 이어서 진행하세요.

목표:
PLURA-XDR 대시보드에서 MITRE ATT&CK 탐지 결과를 확인하고,
각 탐지 행의 상세 화면에서 다음 3가지를 수집합니다.

- 분석
- 로그
- 포렌식

포렌식에서는 이전 값과 현재 값을 비교하여 diff를 생성합니다.

최종 결과는 다음 형식으로 정리해야 합니다.

- Host IP
- 탐지 필터명
- 위험도
- 분석 결과
- 원본 로그 요약
- 포렌식 diff 요약
- Gmail 발송용 제목
- Gmail 발송용 본문
- JSON payload

주의사항:
- 실제 화면 데이터만 사용하세요.
- 예시 HostIP를 사용하지 마세요.
- 날짜는 KST 기준 오늘을 UTC 범위로 변환하세요.
- 검색식 예시는 다음 형식을 따릅니다.
  CreateDate:[2026-04-25T15:00:00Z TO 2026-04-26T14:59:59Z] AND HostIP:<actual-host-ip>
- 모든 실행 결과는 evidence 폴더에 저장하세요.
- 실패 시 어느 단계에서 실패했는지 JSON으로 남기세요.

먼저 현재 상태 요약부터 출력한 뒤, 필요한 최소 수정만 진행하세요.
```

## 핵심은 이것입니다

Codex는 새 스레드로 가도 됩니다.
다만 **이전 대화 전체를 다시 붙이면 또 터집니다.**

대신 다음 3개만 넘기면 됩니다.

1. `README-CODEX-HANDOFF.md`
2. 현재 저장소 파일
3. 다음 작업 프롬프트

이 방식이 가장 안정적입니다.
